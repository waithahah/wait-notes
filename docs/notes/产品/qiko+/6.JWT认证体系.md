# JWT 认证
JWT（JSON Web Token）是一种开放标准（RFC 7519），用于在各方之间安全地传输信息，作为JSON对象。
它被设计为紧凑且自包含的方式，特别适用于分布式站点的用户认证与授权场景，尤其是跨域请求。
JWT允许信息如用户身份在服务器之间安全地无状态（stateless）传递。

## JWT的基本结构
* JWT主要由三部分组成，通过 . 分隔：
    * Header（头部）：描述JWT的元数据，通常包括签名使用的算法类型（如HMAC SHA256或RSA）。
    * Payload（载荷）：包含声明（claims），即用户的相关信息，可以是公开的也可是私有的。这部分信息是经过加密但非密文的，因此不适宜存放敏感信息。
    * Signature（签名）：服务器根据头部指定的算法，结合头部、载荷以及一个秘钥（secret）或公钥，生成一个签名，确保JWT没有被篡改。
* JWT的工作流程
    * 认证：用户向服务器提供凭证（如用户名和密码）进行登录。
    * 服务器验证：服务器验证凭证后，生成一个JWT，其中包含用户标识以及其他声明。
    发送JWT给客户端：服务器将JWT返回给客户端，客户端通常将其存储在本地（如Cookie或LocalStorage）。
    * 客户端携带JWT访问受保护资源：之后的每一次请求，客户端都会附带这个JWT，通常放在HTTP请求头的Authorization字段中。
    * 服务器验证JWT：服务器收到JWT后，会验证其签名和有效期，确认无误后才允许访问受保护的资源。
* JWT的优势
   * 无状态：服务器不需要存储JWT，从而减轻了服务器负担，易于扩展。
   * 安全性：签名机制保证了JWT的不可篡改性。
   * 跨域支持：因为JWT是自包含的，所以可以在不同域名间传递，适用于分布式应用。
   * 到期自动失效：JWT可以设置过期时间，增加安全性。JWT作为一种轻量级的认证和授权方案，在现代Web服务和移动应用中得到了广泛应用。

* 加入租户关联表 
* 加入权限表
* 生成token逻辑
```python
    @classmethod
    def generate_invite_token(cls, tenant: Tenant, account: Account) -> str:
        token = str(uuid.uuid4())
        invitation_data = {
            'account_id': account.id,
            'email': account.email,
            'workspace_id': tenant.id,
        }
        expiryHours = current_app.config['INVITE_EXPIRY_HOURS']
        redis_client.setex(
            cls._get_invitation_token_key(token),
            expiryHours * 60 * 60,
            json.dumps(invitation_data)
        )
        return token
```
## 邀请后激活账号
### 接口地址
`/console/api/activate/check?email=xujl7%40test.com&token=2df12b7c-83ba-46ac-a87b-70635dd93711`

### 逻辑
```python
class ActivateCheckApi(Resource):
    def get(self):
        parser = reqparse.RequestParser()
        parser.add_argument('workspace_id', type=str, required=False, nullable=True, location='args')
        parser.add_argument('email', type=email, required=False, nullable=True, location='args')
        parser.add_argument('token', type=str, required=True, nullable=False, location='args')
        args = parser.parse_args()

        workspaceId = args['workspace_id']
        reg_email = args['email']
        token = args['token']

        invitation = RegisterService.get_invitation_if_token_valid(workspaceId, reg_email, token)

        return {'is_valid': invitation is not None, 'workspace_name': invitation['tenant'].name if invitation else None}

    @classmethod
    def get_invitation_if_token_valid(cls, workspace_id: str, email: str, token: str) -> Optional[dict[str, Any]]:
        invitation_data = cls._get_invitation_by_token(token, workspace_id, email)
        if not invitation_data:
            return None

        tenant = db.session.query(Tenant).filter(
            Tenant.id == invitation_data['workspace_id'],
            Tenant.status == 'normal'
        ).first()

        if not tenant:
            return None

        tenant_account = db.session.query(Account, TenantAccountJoin.role).join(
            TenantAccountJoin, Account.id == TenantAccountJoin.account_id
        ).filter(Account.email == invitation_data['email'], TenantAccountJoin.tenant_id == tenant.id).first()

        if not tenant_account:
            return None

        account = tenant_account[0]
        if not account:
            return None

        if invitation_data['account_id'] != str(account.id):
            return None

        return {
            'account': account,
            'data': invitation_data,
            'tenant': tenant,
        }

    @classmethod
    def _get_invitation_by_token(cls, token: str, workspace_id: str, email: str) -> Optional[dict[str, str]]:
        if workspace_id is not None and email is not None:
            email_hash = sha256(email.encode()).hexdigest()
            cache_key = f'member_invite_token:{workspace_id}, {email_hash}:{token}'
            account_id = redis_client.get(cache_key)

            if not account_id:
                return None

            return {
                'account_id': account_id.decode('utf-8'),
                'email': email,
                'workspace_id': workspace_id,
            }
        else:
            data = redis_client.get(cls._get_invitation_token_key(token))
            if not data:
                return None

            invitation = json.loads(data)
            return invitation
```

# 问题
1. 自动续期问题？
JWT本身不具备自动续期的功能，一旦设置的过期时间到达，该JWT就会过期失效。如果用户在过期时间内没有进行任何操作，再次尝试使用该JWT访问受保护资源时，服务器会拒绝请求，这时用户看起来就像是被“退出”了登录状态。

为了提升用户体验并避免频繁登录，通常会采用以下策略之一或结合使用来解决JWT过期问题：

刷新令牌（Refresh Token）：

* 在用户首次登录成功时，除了返回一个短时效的JWT（例如你提到的3小时过期的access token），还会返回一个长时效的刷新令牌（refresh token）。
当前端检测到JWT即将过期或已经过期时，可以静默使用refresh token向后端请求一个新的JWT，而无需用户重新输入用户名和密码。
Refresh token通常有更严格的存储要求和更长的有效期，且应当通过安全通道传输。
定时刷新JWT：

* 前端可以在用户活跃期间（比如每次发起请求前），检查JWT的剩余有效期。如果接近过期时间（比如还剩10分钟），前端可以主动请求后端刷新JWT，延长会话时间。
这种方式需要后端支持刷新逻辑，并且前端需合理设计刷新逻辑以避免不必要的刷新请求。
长会话与滑动窗口：

*  实现一个滑动窗口机制，即每当用户有活动（如API请求），就更新JWT的有效期，让它始终维持在一个相对固定的有效时间段内（比如总是保持接下来3小时内有效）。
这样做可以让用户的登录状态在持续活跃时保持有效，但需要后端支持并记录最后活跃时间，以决定是否延长JWT的有效期。
综上所述，通过上述策略可以在一定程度上确保用户在活跃使用时不会因为JWT过期而被强制退出登录。
