1. 实时因子（Real-Time Factor, RTF）
定义：
RTF 表示处理一段语音所需的时间与实际语音时长的比值。公式为：

RTF < 1：处理速度快于实时（例如，处理10秒语音耗时5秒，RTF=0.5）。

RTF = 1：处理速度与实时同步。

RTF > 1：处理速度慢于实时（例如，处理10秒语音耗时15秒，RTF=1.5）。

意义：

衡量系统的实时性，尤其在流式识别（如电话客服、实时字幕）中至关重要。

RTF越低，系统响应越快，用户体验越好。

影响因素：

模型复杂度（如深度学习模型的计算量）。

硬件性能（CPU/GPU算力）。

音频预处理（降噪、分帧等）和后处理（语言模型解码）的耗时。

2. 吞吐量（Throughput）
定义：
吞吐量指系统在单位时间内能处理的语音数据量，通常以以下两种方式表示：

小时/秒（Hours Processed per Second, HPS）：每秒可处理多少小时的语音。

并发流数：同时处理的音频流数量（如“支持100路并发”）。

意义：

反映系统的批量处理能力，适合离线场景（如批量转写大量录音）。

高吞吐量意味着能高效处理大规模任务。

影响因素：

硬件并行能力（多核CPU/GPU、分布式计算）。

模型优化（如批处理推理）。

内存和I/O速度（数据读取、传输效率）。

## 降噪
1. SNR: 13.30 dB
信噪比（Signal-to-Noise Ratio）

13 dB 意味着语音能量比噪声大约 20 倍左右（10^(13/10) ≈ 20）

常见参考：

> 20 dB：几乎无噪声，音质好

10~20 dB：可理解，但有明显噪声

< 10 dB：噪声显著，听感差

你的 13.3 dB → 属于中等，有一定噪声，但不至于严重影响识别

2. Flatness: 0.44
频谱平坦度（Spectral Flatness Measure, SFM）

衡量信号是像“纯音”还是“噪声”：

0 → 纯音（比如哨声）

1 → 白噪声（均匀分布的能量）

0.44 → 说明有一定的噪声成分，但不是纯噪声，属于混合型语音

3. RMS: 0.1204
均方根幅度，反映音量大小

这个值没有固定好坏，要结合系统的参考范围

0.12 在很多归一化到 [-1,1] 的音频里算中等音量，不需要特别处理

4. Noise Prob: 0.00
噪声概率，可能来自某种噪声检测器

0.00 表示几乎没有检测到噪声片段（要注意这个值可能是阈值判定结果，而非绝对真值）

5. Score: 2.00
可能是整体质量评分（不同工具标准不一样，满分可能是 3、4 或 5）

如果满分是 3 → 2 分算合格

如果满分是 5 → 2 分就偏低

一、判定分级（默认阈值，工程可直接用）
先给结论：按三档来决定处理方式。

A. 直接识别（不做深度降噪）
满足全部（或多数）条件即可：

SNR ≥ 18 dB

Flatness ≤ 0.50

NoiseProb ≤ 0.20

0.03 ≤ RMS ≤ 0.60（音量不过小、也不过大/削顶）

质量 Score ≥ 2/3 满分（若你的满分是 5，则 ≥ 3）

可选：做轻量前端（增益规范化、门限降噪、VAD 裁剪），不跑深度模型。

B. 轻量降噪后识别（实时/低开销）
出现下列任一情况（但没到“严重噪声”）：

12 dB ≤ SNR < 18 dB

0.50 < Flatness ≤ 0.70

0.20 < NoiseProb ≤ 0.50

RMS < 0.03（太小）或 RMS > 0.60（可能失真）

Score 略低（如 2/5、2/4）

建议：RNNoise / WebRTC-NS 这类轻量抑噪 + 正常 ASR；或直接用 ASR 内置前端。

C. 深度降噪后识别（离线/可以牺牲时延）
满足任一严重条件：

SNR < 12 dB

Flatness > 0.70

NoiseProb > 0.50

RMS 极端（< 0.01 或 > 0.90，明显过暗/削顶）

Score 明显偏低（如 1/5）

建议：Demucs/DTLN/Conv-TasNet/VoiceFixer 等深度降噪，离线处理再识别。
