为什么 1.7B 模型更适合低设置？
因为 Qwen3-1.7B 本身非常小（只占 3.5GB），即使你只给它 0.6 的显存，剩下的 KV Cache 空间（约 11GB）依然能撑起 120+ 的并发（因为每个 10s 请求只需几十 MB 的 KV Cache）。

既然 120+ 并发已经满足了你的业务需求，那么多余的显存留在“墙外”做计算工作区，收益远比留在“墙内”当空闲车位要高。

这就是为什么你发现 0.6 或 0.75 性能最好。

这两个参数的关系，就像是**“土地总量”与“规划车位数”**的关系。它们共同决定了你的 GPU 能同时接多少活，以及干活有多快。简单来说：GPU_MEMORY_UTILIZATION 决定了显存里划出多大一块地来停“车”（KV Cache），而 MAX_NUM_SEQS 则是你在这块地上强行划了多少个“车位”。1. 它们之间的“物理公式”在 vLLM 内部，它们的关系遵循这个逻辑：$$\text{KV Cache 总空间} = (\text{显存总量} - \text{模型权重} - \text{系统预留}) \times \text{GPU\_MEMORY\_UTILIZATION}$$$$\text{理论最大并发数} = \frac{\text{KV Cache 总空间}}{\text{max\_model\_len} \times \text{单 Token 显存开销}}$$MAX_NUM_SEQS 只是个“上限门禁”：如果你设了 150，但显存利用率设得很低，导致 KV Cache 空间只够停 50 辆车，那么实际并发依然只有 50。显存报错（OOM）的由来：如果你把 MAX_NUM_SEQS 设得很大（比如 512），而 max_model_len 也很大（比如 20000），那么公式算出来的空间会远超显存总量，模型直接启动失败。2. 深度博弈：为什么 100 并发和 150 并发时间一样？这是你实验中最核心的发现。让我们看看它们是怎么互相牵制的：场景 A：利用率 0.9 + 并发 100 → 115秒土地分配：KV Cache 占了 90% 的地，留给计算（Workspace）的地只有 10%。后果：虽然你划了 100 个车位，但因为“过道”太窄，音频编码器在算 100 个请求的特征时，显存频繁发生拥堵和交换。GPU 核心空有 100% 的力气，却因为没地方落脚而只能慢吞吞地干。场景 B：利用率 0.6 + 并发 100 → 63秒土地分配：KV Cache 只占 60%，留给计算的空间暴增到 40%。后果：过道变宽了。100 个请求进来，音频编码器可以瞬间展开，并行计算。这证明了：对于 Qwen3-ASR，计算空间（Workspace）的优先级高于存储空间（KV Cache）。场景 C：利用率 0.75 + 并发 150 → 63秒土地分配：平衡了存储和计算。后果：此时虽然车位多了（150个），过道也够宽，但厨师（GPU 算力）忙不过来了。4090 的核心每秒钟处理的语音帧数已经达到了物理极限。即使你再加车位，也只是让更多音频在门口排队，总的“完工时间”不再缩短。3. 它们如何配合才是最优？调优这两个参数的正确姿势应该是：“先定车位，再留过道”。先看并发需求 (MAX_NUM_SEQS)：你根据 Redis 队列的压力，确定自己需要 100 还是 120 的并发。再算 max_model_len：因为是 10s 短音频，固定在 1024。最后调 GPU_MEMORY_UTILIZATION：通过降低这个值，主动释放 KV Cache，把显存还给 Workspace。黄金准则：只要不报 Maximum concurrency... is less than MAX_NUM_SEQS 的错误，这个值越低，推理速度通常越快。4. 总结MAX_NUM_SEQS 是你想跑多少路。GPU_MEMORY_UTILIZATION 是你准许划走多少显存来存数据。你现在的 63 秒意味着：在 max_model_len=1024 的前提下，100 个车位已经把 4090 的算力填满了。你把利用率降到 0.6 或 0.75，是给计算让出了路；但路顺了之后，车速就受限于发动机（CUDA 核心）本身了。我有个大胆的猜测： 如果你把 MAX_NUM_SEQS 降到 50，处理那 440 个片段的时间可能会变成 100秒 以上；如果你保持 100 但把利用率升到 0.95，时间会变成 130秒 以上。现在的 0.7 显存 + 120 并发 就是你的“甜点位”。你觉得这个识别速度在你的业务里够用了吗？
