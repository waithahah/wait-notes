# 文件上传
这段代码并不是一个完整的函数，而是在某个上下文中对已读取文件操作结果的处理部分。为了帮助您理解从文件读取到获取Base64编码的整个过程，我将简要说明涉及的几个关键步骤，虽然这超出了原始提问的范围，但希望能解答您的疑惑。

文件选择：用户通过HTML的<input type="file">元素选择一个文件。

FileReader API：当文件被选中后，JavaScript通常会使用FileReader API来异步读取这个文件的内容。FileReader提供了几个方法，其中最常用的是readAsDataURL方法，它能够读取指定的Blob或File对象，并将其内容读取为一个data URL，而data URL是一种特殊格式的字符串，常用于嵌入图片、音频等资源到文档中，其前缀是data:，后面跟着MIME类型，然后是base64编码的数据。

事件监听：调用readAsDataURL后，不会立即得到结果，因为文件读取是异步的。因此，需要给FileReader添加一个事件监听器，如onload事件，当文件读取完成时，这个事件会被触发。

转换为Base64：在onload事件处理器中，reader.result将会包含文件的base64编码数据，这是因为我们使用了readAsDataURL方法。此时，reader.result是一个包含了文件内容的data URL字符串，而实际的Base64编码是这个字符串中data:之后的部分。

类型断言：在您提供的代码片段中，reader.result as string是用来告诉TypeScript编译器，尽管reader.result可能有多种类型（根据不同的读取方法），但我们明确知道在这里它应该被当作字符串类型处理。这样就可以安全地将它赋值给预期为字符串类型的变量base64Url。

综上所述，通过FileReader的readAsDataURL方法读取文件，并在加载完成后通过事件处理回调，可以获取到文件内容的Base64编码字符串，代码中的类型断言只是确保了数据类型的正确处理。

```python
import jwt
from flask import current_app
from werkzeug.exceptions import Unauthorized


class PassportService:
    def __init__(self):
        self.sk = current_app.config.get('SECRET_KEY')
    
    def issue(self, payload):
        return jwt.encode(payload, self.sk, algorithm='HS256')
    
    def verify(self, token):
        try:
            return jwt.decode(token, self.sk, algorithms=['HS256'])
        except jwt.exceptions.InvalidSignatureError:
            raise Unauthorized('Invalid token signature.')
        except jwt.exceptions.DecodeError:
            raise Unauthorized('Invalid token.')
        except jwt.exceptions.ExpiredSignatureError:
            raise Unauthorized('Token has expired.')
```

JWT，全称为JSON Web Token，是一种开放标准（RFC 7519），提供了一种紧凑且自包含的方法，用于在各方之间安全地传输信息。这个信息通常以JSON对象的形式存在，并且因为经过数字签名，所以可以被验证和信任，确保其完整性和来源的真实性。

简单来说，JWT充当了一种安全令牌，可以在客户端与服务器之间安全地传输，用来验证用户身份、进行授权等。它主要由三部分组成：头部（Header）、载荷（Payload）和签名（Signature）。JWT的工作流程一般包括以下步骤：

签发：服务器根据用户凭证生成一个JWT，其中可以包含用户ID、角色、过期时间等声明。
传输：JWT通过HTTP请求头或其他方式发送给客户端，客户端可以存储JWT（例如在Cookie或本地存储中）。
验证：客户端在后续请求中附带JWT，服务器端接收后验证其签名和有效性（如检查过期时间），从而确认用户身份并授权访问资源。
JWT的优势包括但不限于：

无状态：服务器无需存储会话信息，降低了服务器的存储需求。
安全性：由于签名机制，JWT难以篡改，保证了信息的安全传输。
跨域支持：适用于分布式系统的不同域间认证。
自包含：所有必要的认证信息都包含在JWT内部，减少了对额外查询数据库的需求。
